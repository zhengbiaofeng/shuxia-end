# 接口联调规范

更新时间：2026-06-07

## 目标

接口联调必须做到路径可追溯、字段有归一化、页面可验证、结果有记录。前端页面不直接依赖后端 VO 字段，接口变更优先收敛在 `src/api/*`。

## 标准流程

1. **确认接口契约**
   - 以 `http://127.0.0.1:18080/jeecg-boot/v3/api-docs` 为准。
   - 记录路径、方法、query/path 参数、request body、response result shape。
   - 发现旧路径或文档不存在的路径，先改 API 层，不在页面里绕过。

2. **探测鉴权边界**
   - 未登录请求：应返回 401。
   - 无效 token 请求：应返回 401。
   - 有效 token 请求：应返回 200，并记录 result 的结构摘要。

3. **编写 API 层**
   - 请求统一放在 `src/api/*`。
   - 每个接口函数负责校验 `response.success`。
   - 写操作统一返回后端 `result`，失败时抛出后端 `message`。

4. **字段归一化**
   - 后端字段转换为前端稳定模型，例如 `sourceName` 转 `name`。
   - 页面和组件只消费归一化后的数据。
   - 格式化逻辑如容量、时间、状态文案集中在 API normalizer 或共享 helper。

5. **页面接入**
   - 页面负责加载、提交、刷新、错误提示。
   - 组件负责展示和触发事件，不直接写接口路径。
   - 列表失败允许使用静态数据兜底，但写操作失败必须明确提示。

6. **验收记录**
   - 每轮联调更新 `docs/api-debug-report.md`。
   - 记录有效 token 下接口状态、页面渲染数量、失败点和后续建议。

## 写操作约定

- 新增、编辑、删除、启禁用都必须刷新当前列表。
- 删除只允许由用户触发确认，不做静默删除。
- 联调测试只创建 `codex-*` 或明确测试命名的数据，并在验证后清理。
- 写接口不在组件内部调用，必须由页面或 store 统一调度。

## 错误处理约定

- 401：清理本地 token 并回登录页，或让请求层错误向上冒泡。
- 业务失败：展示后端 `message`。
- 网络失败：展示通用失败提示，并保留当前页面已有数据。
- 控制台日志需包含模块名，方便定位，例如 `标签保存失败`。
